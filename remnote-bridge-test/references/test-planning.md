# 测试规划指南

每次有新功能或修改时，按此指南动态规划测试方案。

---

## 1. 分析变更

首先搞清楚改了什么、影响了什么。

### 变更分类

| 变更类型 | 示例 | 测试重点 |
|:---------|:-----|:---------|
| **新命令/工具** | 新增 `batch-edit` 命令 | 核心功能 + 参数组合 + 错误处理 |
| **现有命令修改** | edit-tree 支持新的操作类型 | 新功能 + 原有功能不退化 |
| **Bug 修复** | 修复 edit-rem 的 tags diff 逻辑 | 复现原 bug + 修复后行为 + 相关场景回归 |
| **文档修改** | 重写 edit-tree instructions | 文档可理解性（haiku 能否仅凭新文档完成操作） |
| **Plugin 修改** | 修改 services/read-tree.ts | 受影响的 CLI 命令端到端测试 |
| **MCP 层修改** | 修改 tool description | MCP subagent 能否正确使用 |

### 影响范围清单

逐一确认：

- [ ] 涉及哪些 CLI 命令？
- [ ] 涉及哪些 MCP 工具？
- [ ] 涉及哪些 Skill instruction 文档？
- [ ] 改了参数/返回值格式吗？
- [ ] 改了错误信息/错误处理吗？
- [ ] 改了缓存行为吗？
- [ ] 改了防线逻辑吗？
- [ ] 有跨命令/跨层的连带影响吗？

---

## 2. 生成测试用例

根据变更类型和影响范围，动态设计测试用例。

### 测试用例设计原则

**从简单到复杂**：先验证核心功能，再测边界情况。如果基础功能都不行，复杂场景测了也白测。

**自生成数据**：每个测试自己在测试页面下创建数据，不依赖预置状态。这确保测试可重复、自包含。

**可验收性**：每个测试用例必须有明确的"Chrome 怎么看、看什么、对才算对"。

### 用例模板

每个测试用例包含：

```
名称：简短描述（如 "edit-tree 新增行带 type 元数据"）
前缀：测试数据命名前缀（如 "[0317-meta]"）
目的：这个测试验证什么
步骤：
  1. 创建什么测试数据
  2. 执行什么操作
  3. 预期结果是什么
Chrome 验证：
  - 具体检查哪些维度（文本/结构/属性/闪卡...）
  - 每个维度的预期状态是什么
```

### 不同变更类型的用例策略

#### 新功能

1. **Happy path**：最基本的正确用法（1-2 个用例）
2. **参数变体**：不同参数组合（按需 1-3 个）
3. **与现有功能组合**：新功能和已有功能一起使用（1-2 个）
4. **边界情况**：空值、极大值、非法值（按需 1-3 个）
5. **错误处理**：预期的错误场景是否返回正确提示（1-2 个）

#### Bug 修复

1. **复现用例**：重现原始 bug 的精确场景
2. **修复验证**：同样场景，预期正确行为
3. **回归用例**：相关功能是否仍正常（1-2 个）

#### 文档修改

1. **核心流程**：文档中描述的主要使用流程能否走通
2. **新增说明**：新增的使用指引是否清晰
3. **易混淆点**：文档中容易误解的部分（如 edit-rem vs edit-tree 的区分）

#### 回归测试

选择受影响命令的核心路径：
1. 该命令最基本的用法
2. 该命令最常用的参数组合
3. 该命令与其他命令的交互（如 read → edit 链路）

## 3. 测试用例的粒度

### 一个好的测试用例

- **一个明确的验证目标**（不是"测试 edit-tree 的所有功能"）
- **自包含**（创建数据 → 操作 → 验证，不依赖其他测试）
- **可验收**（Chrome 能看到什么、看什么维度、什么算对）
- **适合低智能模型执行**（步骤清晰，不需要复杂推理）

### 粒度参考

| 太粗 ❌ | 合适 ✅ | 太细 ❌ |
|:--------|:--------|:--------|
| "测试 edit-tree" | "edit-tree 创建 3 个子节点并验证" | "edit-tree 创建 1 个节点" |
| "测试闪卡功能" | "创建正向+反向+双向闪卡并验证方向" | "创建 1 个正向闪卡" |
| "测试所有属性" | "修改 type/highlightColor/fontSize 并验证" | "修改 type 为 concept" |

---

## 4. 复杂场景规划

当需要测试极端或复杂场景时（如用户说的"30 个工具 20 个移动 10 个修改混杂"）：

### 批量操作测试

```
目的：验证大规模混合操作的可靠性
步骤：
  1. 批量创建 N 个节点
  2. 批量移动 M 个节点（改变父子关系）
  3. 批量修改 K 个节点（不同属性）
Chrome 验证：
  - 所有节点存在
  - 层级关系全部正确
  - 属性全部生效
```

### 参数规模

| 规模 | 创建 | 移动 | 修改 | 适用场景 |
|:-----|:-----|:-----|:-----|:---------|
| 小 | 5 | 3 | 2 | 日常功能验证 |
| 中 | 15 | 10 | 5 | 功能稳定性验证 |
| 大 | 30 | 20 | 10 | 压力测试/发布前验证 |

规模根据实际需要选择，不是每次都要跑最大的。

---

## 5. 规划输出

规划完成后，输出格式：

```
## 测试计划：{功能名称}

### 变更摘要
- 改了什么、影响什么

### 测试用例
1. {用例名} — MCP + Skill
   - 步骤：...
   - Chrome 验证：...
2. {用例名} — MCP + Skill
   - ...

### 执行顺序
基础功能 → 参数变体 → 组合场景 → 边界情况

### 预计 subagent 数量
{N} 个用例 × 2 接口 = {2N} 轮
```

规划获得用户确认后，进入执行阶段（读 `subagent-templates.md` 构造 prompt）。
