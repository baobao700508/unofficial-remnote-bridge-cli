# 回归测试基线

固定的回归测试用例集。按级别组织，前序级别不过则后续全部跳过。

**本文件是活的**：随着功能增加，往对应级别里追加新用例。

---

## 使用方式

### 全量回归（发布前）

从 L1 逐级跑到 L5，全部 PASS 才算通过。

### 快速回归（小改后）

只跑 L1 + 受影响级别。例如改了 edit-tree → 跑 L1 + L2(创建/删除) + L3(移动重排)。

### 选择性回归

用户指定跑哪些用例。

---

## L1 基础设施

> 连接不通一切白搭。

### L1-01: 连接生命周期

**task_description**:
```
检查 RemNote 工具的连接状态。

1. 检查系统健康状态
2. 如果未连接，建立连接
3. 再次检查健康状态，确认 daemon / Plugin / SDK 三层都正常
4. 断开连接
5. 检查健康状态，确认已断开
```

**验证**：无需 Chrome（纯基础设施）。检查 subagent 是否正确理解三层依赖、正确执行生命周期。

**断言**：
- [ ] 正确调用 health
- [ ] 正确解读三层状态
- [ ] connect/disconnect 成功

---

### L1-02: 知识库探索

**task_description**:
```
探索 RemNote 知识库的整体结构。

1. 确认连接正常
2. 获取知识库的全局文档概览
3. 从结果中找到名为"MCP 测试"的文档，报告它的 remId
```

**验证**：subagent 能从 read_globe 输出中正确解析文档名和 remId。

**断言**：
- [ ] read_globe 成功
- [ ] 正确识别"MCP 测试"文档
- [ ] 提取的 remId 正确

## L2 基础读写

> 单操作验证。

### L2-01: 创建 + 读取

**task_description**:
```
在测试页面下创建 3 个子节点并确认。

1. 确认连接正常
2. 读取测试页面的子树
3. 创建 3 个新子节点：
   - "{prefix} 节点 A"
   - "{prefix} 节点 B"
   - "{prefix} 节点 C"
4. 重新读取子树，确认 3 个节点都出现
```

**Chrome 验证**：
- ⬜ 3 个节点存在，文本完全正确，顺序正确

---

## L3 工作流

> 多步组合链路。

### L3-01: 搜索 → 读取 → 编辑

**task_description**:
```
搜索定位后编辑内容。

1. 确认连接正常
2. 读取测试页面子树
3. 创建节点：{prefix} uniquetoken_abc123
4. 搜索 "uniquetoken_abc123"
5. 从结果中找到刚创建的节点
6. 读取它的详细属性
7. 修改文本为 "{prefix} 搜索后已修改"
8. 读取确认
```

**Chrome 验证**：
- ⬜ 修改后的文本正确

---

### L3-02: 知识结构（概念 + 闪卡）

**task_description**:
```
创建概念-描述知识结构。

1. 确认连接正常
2. 读取测试页面子树
3. 创建以下结构：
   - "{prefix} 数据结构"（概念类型，双向闪卡，背面："计算机中组织和存储数据的方式"）
     - "{prefix} 数组"（正向闪卡，背面："连续内存的线性结构"）
     - "{prefix} 链表"（正向闪卡，背面："非连续内存的链式结构"）
4. 读取创建结构并验证所有类型、闪卡方向、backText
```

**Chrome 验证**：
- ⬜ "数据结构"加粗（concept）+ 双向箭头 + 正确 backText
- ⬜ "数组""链表"有正向箭头 + 正确 backText
- ⬜ 层级正确（数组/链表是数据结构的子节点）

---

### L3-03: 移动 + 重排

**task_description**:
```
创建节点后移动和重排。

1. 确认连接正常
2. 读取测试页面子树
3. 创建 3 个同级节点：{prefix} A、{prefix} B、{prefix} C
4. 重新读取子树
5. 将 C 移动为 A 的子节点
6. 将 B 排到 A 前面
7. 读取最终结构确认
```

**Chrome 验证**：
- ⬜ C 是 A 的子节点
- ⬜ B 排在 A 前面

## L5 极端边界

> 大规模操作、错误恢复、边界条件。

### L5-01: 30创建 + 20移动 + 10修改

**task_description**:
```
大规模混合操作。

1. 确认连接正常
2. 读取测试页面子树

A. 创建 30 个子节点：{prefix} N-001 到 N-030
B. 将 N-021~N-030 移到 N-001 下面
C. 将 N-011~N-020 移到 N-002 下面
D. N-001~005 设为 concept
E. N-006~010 设为不同颜色高亮（Red/Orange/Yellow/Green/Blue）

3. 用足够大的 maxNodes 读取完整结构确认
```

**Chrome 验证**：
- ⬜ 30 个节点全在
- ⬜ 层级正确（N-001 下有 10 子、N-002 下有 10 子）
- ⬜ 属性正确（加粗、颜色）

---

独立测试每个场景：

## 维护指南

### 新增回归用例

当新功能稳定后，把它的测试用例加到对应级别里：

1. 确定属于哪个级别（基础→复杂）
2. 编写 task_description（参照已有格式）
3. 列出 Chrome 验证点
4. 加到对应级别下，编号递增

### 退役用例

功能被移除时，把对应用例标记为 `[RETIRED]` 并注明原因，而非直接删除。

### {prefix} 替换

执行时把 `{prefix}` 替换为 `[日期-级别编号]`，如 `[0317-L2-01]`。
